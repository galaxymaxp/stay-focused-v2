## Encryption Design

- API environment variable: `CANVAS_TOKEN_ENCRYPTION_KEY`
- Required key format after Phase 5A.2: canonical padded Base64-encoded value;
  decoded key must be exactly 32 bytes. Surrounding environment whitespace is
  accepted for the key only.
- Ownership: application/deployment-owned Stay Focused secret, not supplied by
  Canvas, the school, or the student.
- Algorithm: AES-256-GCM through Node `crypto`.
- Token storage fields: ciphertext, IV, authentication tag, and encryption
  version.
- Decryption fails closed for corrupted ciphertext, corrupted authentication
  tag, unsupported version, and invalid key.
- Stored ciphertext, IV, and authentication-tag fields are decoded as strict
  canonical Base64; IV must decode to 12 bytes and authentication tag to 16
  bytes.
- The key and encrypted payload fields are validated when encryption or
  decryption is used, not at process startup.

## Authentication Model

- Phase 5A uses a personal access token generated by each individual Canvas
  user.
- There is no school-wide Canvas token and no shared application Canvas token.
- Each token belongs only to the Canvas user who generated it and inherits only
  that user's Canvas permissions.
- Tokens cannot access school-wide data or bypass locked, hidden, unpublished,
  or permission-restricted content.
- The developer-owned `CANVAS_PERSONAL_ACCESS_TOKEN` value is used only by the
  direct live-validation harness.
- Normal application behavior accepts the authenticated user's submitted Canvas
  base URL and personal access token, validates it as that user, encrypts it
  server-side, stores it for that Stay Focused user only, and later uses only
  that stored per-user credential.
- OAuth is not implemented in Phase 5A. It remains the intended production
  authorization path for broad multi-user deployment after an
  institution-approved Canvas Developer Key is available.
